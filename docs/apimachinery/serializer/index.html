序列化器与序列化器工厂

在本节中我们将深入介绍一下apimachinery库中的序列化器和序列化工厂。

note

对于apimachinery库中序列化器的编/解码流程，我们不会照搬大段源码并做行级别的解释。它的实现为了考虑各种情况远比我们想象的要复杂。 本节介绍的序列化内容仍然是为了服务于本书的目标——能够理解最后实现的精简自定义控制器代码。

JSON序列化器​

在之前的小节中，我们引入了序列化器的概念，并且知道apimachinery库中的序列化器需要依赖scheme才能工作。 接下来我们从源码的角度先介绍apimachinery库中最为基础的序列化组件JSON序列化器的解码流程。

k8s.io/apimachinery/pkg/runtime/serializer/json/json.go
type Serializer struct {
    meta    MetaFactory
    options SerializerOptions
    creater runtime.ObjectCreater
    typer   runtime.ObjectTyper

    identifier runtime.Identifier
}

我们将重点关注meta，creater，typer这三个编解码过程中涉及到的核心成员类。 你可能发现并没有与runtime.Scheme类相关的成员类，那么这个基础的JSON序列化组件难道不需要scheme就可以工作吗？ 事实并非如此，正如前文所说，JSON序列化器是apimachinery库中最为基础的组件， 官方开发者特意在实现上做了更进一步的抽象——它并不直接依赖于runtime.Scheme对象，而是任何与runtime.Scheme有同等能力的类型都可以。 具体来说，只要实现了runtime.ObjectCreater和runtime.ObjectTyper两个接口的类型都可以，其中：

  • runtime.ObjectTyper所要表述的能力是：给定一个runtime.Object对象，判定此对象是否被认可（recognize）1；
  • runtime.ObjectCreater所要表述的能力是：给定一个GVK三元组信息，能够创建此GVK对应Go类型的实例；

而runtime.Scheme的确实现了runtime.ObjectCreater和runtime.ObjectTyper接口。 并且，apimachinery库中在初始化Serializer对象的成员变量creater和typer时使用的确实是一个scheme对象。

meta的作用是从原始JSON字节序列（bytes[]）中强行推理（interpret）出此字节序列包含的GVK信息。具体实现上也十分简单清晰：

k8s.io/apimachinery/pkg/runtime/serializer/json/meta.go
func (SimpleMetaFactory) Interpret(data []byte) (*schema.GroupVersionKind, error) {
    findKind := struct {
        // +optional
        APIVersion string `json:"apiVersion,omitempty"`
        // +optional
        Kind string `json:"kind,omitempty"`
    }{}
    if err := json.Unmarshal(data, &findKind); err != nil {
        return nil, fmt.Errorf("couldn't get version/kind; json parse error: %v", err)
    }
    gv, err := schema.ParseGroupVersion(findKind.APIVersion)
    if err != nil {
        return nil, err
    }
    return &schema.GroupVersionKind{Group: gv.Group, Version: gv.Version, Kind: findKind.Kind}, nil
}

在函数中预先声明了一个仅包含APIVersion以及Kind的空结构体，尝试先将字节序列解码为此结构来获取此字节序列的原始gvk的信息。

findKind := struct {
    // +optional
    APIVersion string `json:"apiVersion,omitempty"`
    // +optional
    Kind string `json:"kind,omitempty"`
}{}

在正式介绍JSON序列化器的解码方法之前，我们先看一下apimachinery库中约定的解码器所需要实现的接口：

type Decoder interface {
    Decode(data []byte, defaults *schema.GroupVersionKind, into Object) (Object, *schema.GroupVersionKind, error)
}

通常，对于一个解码函数，它的签名应该类似于Decode(data []byte) (Object, error)，它的作用顾名思义就是把一段字节序列转化为一个Go对象。

而在apimachinery库中约定的Decode()方法中，除了原始字节序列数据data这个参数以外，还需要defaults以及into两个参数。解码方法这样设计的原因是提供了更多的灵活性。 官方开发者希望解码器能够：

  1. 通过原始字节序列（data）包含的gvk信息以及调用者提供的gvk信息（defaults）共同推断出一个可能的gvk2；
  2. 如果调用者已经预先提供了解码后的目标对象"载体"（into），解码器应该尽量使用调用者提供的载体而避免在内存中再分配一个新对象3。

接下来我们会大致4叙述一下JSON序列化器的解码流程。 正如前文所述，由于apimachinery库中约定的Decode()方法需要额外提供一些灵活性，导致最终解码后得到的Go对象可能是一个推断出的gvk。 JSON序列化器在实现上也并不例外。 在JSON序列化器的解码方法中，推断gvk的具体顺序为： originalData > default gvk > into，也就是原始字节数据包含的gvk信息优先于调用者给定的gvk信息（参数defaults）优先于参数中事先提供的对象载体（参数into）。

  1. 首先通过meta.Interpret()方法尝试从原始字节数据中得到gvk信息，我们称此gvk信息为实际gvk；
  2. 接下来通过gvkWithDefaults()函数将实际gvk结合提供的默认gvk（参数defaults）再推断出一个gvk，我们称此gvk信息为推断gvk；
  3. 如果推断gvk与提供的目标对象载体（参数into) 的gvk仍然不匹配，那么在runtime.UseOrCreateObject()函数中创建一个新的与推断gvk匹配的对象载体，之后将原始字节数据反序列化到此新载体上并返回此新载体； 否则，直接将原始字节数据反序列化到调用者提供的目标载体（参数into)中并返回此目标载体。

相比于解码流程，编码流程相当简单，我们在这里不再做赘述。

序列化器工厂CodecFactory​

在apimachinery库中，上述的JSON序列化器其实是一个基础组件，通常在Kubernetes中我们使用的是更上层的序列化器封装CodecFactory：

type CodecFactory struct {
    scheme    *runtime.Scheme
    universal runtime.Decoder
    accepts   []runtime.SerializerInfo

    legacySerializer runtime.Serializer
}

CodecFactory实现了NegotiatedSerializer5这个核心接口:

k8s.io/apimachinery/pkg/runtime/interfaces.go
type NegotiatedSerializer interface {
    SupportedMediaTypes() []SerializerInfo

    EncoderForVersion(serializer Encoder, gv GroupVersioner) Encoder
    DecoderToVersion(serializer Decoder, gv GroupVersioner) Decoder
}
  1. SupportedMediaTypes()表示实现类需要同时支持多种数据格式（例如JSON，YAML，protobuf）的编解码，换句话说，实现类需要集成多种数据格式的序列化器。
  2. EncoderForVersion()以及DecoderToVersion()表示实现此接口的工厂类需要提供可以将资源编/解码成某个指定版本的编解码器——将资源在解码后或者编码前*转换（convert）*成指定版本（version），例如将某资源的v1beta1版本转换为v1版本3。 需要注意的是实现类CodecFactory同时也提供了WithoutConversion()方法以获取不带资源版本转化功能的编解码器。🎈

我们再稍微介绍CodecFactory类中的一些重要成员:

  • scheme成员——序列化器在编解码前需要检查scheme中注册的类型。
  • accepts切片包含多种数据格式的序列化器，上述的JSON序列化器只是其中的一种。
  • universal成员的作用是提供了一个通用快捷的解码器，它能够识别字节序列的数据格式（例如判定数据是属于JSON还是YAML）再找到对应的序列化器进行解码。

ParameterCodec​

上述的CodecFactory主要用于HTTP请求/返回体以及写入/读取etcd的编解/码。在Kubernetes中，还需要一种特殊的"编解码器"6，它的作用是负责HTTP URL参数（Query Parameters）与Go对象之间的转换。 例如，Go对象metav1.ListOptions{Limit: 1, LabelSelector: "app=foo"}7将会解码成URL参数limit=1&app%3D=foo。 另外，值得注意的是，ParameterCodec编解码器仍然依赖于Scheme注册机制工作。也就是说，ListOptions类型也同样需要事先注册进scheme中。

小结​

tip

本节介绍的apimachinery库中CodecFactory以及ParameterCodec两个类型在我们实现自定义控制器的客户端有着重要应用。

  1. 认可（recoginize）在这里是一种对应接口层级的抽象表述。只有被认可的对象才肯以被序列化。Scheme的注册即是一个具体实现认可的机制。↩

  2. 这样设计的一个好处是对于原始数据未包含gvk信息的情况（空字符串），我们可以通过调用者提供的gvk信息（defaults）甚至是into的gvk信息兜底。 当然，这种灵活性也带来了实现上的不方便。尤其是原始数据包含的实际gvk信息和调用者提供的gvk信息以及into的gvk不一致的情况。 因此在编码器接口的实现中（例如JSON序列化器），有相当一部分逻辑在处理这种情形。↩

  3. 资源不同版本的转化是Kubernetes服务端处理资源请求的重要部分，通过将资源转化成一个内部版本(internal/hub version)以便对不同版本的资源统一进行相关逻辑的处理。 我们在这里仅给出一个Kubernetes服务端资源版本转换发生的一般流程（改编自文档API Changes ）：

    1. A user POSTs a Pod object to /api/v7beta1/...
    2. The JSON is unmarshalled into a v7beta1.Pod structure
    3. Default values are applied to the v7beta1.Pod
    4. The v7beta1.Pod is converted to an hub.Pod internal version
    5. The hub.Pod is validated, and any errors are returned to the user
    6. The hub.Pod is converted to a v6.Pod (because v6 is the latest stable version)
    7. The v6.Pod is marshalled into JSON and written to etcd

    相关内容不在本书讨论范围之内，请参考Kubernetes API请求的生命周期，以及Kubernetes资源的版本转化 apimachinery库中实现了一个带有资源版本转换功能的编解码器codec(k8s.io/apimachinery/pkg/runtime/serializer/versioning)，对于编/解码功能，它仍然基于的是基础序列化器。 由于所有资源类型的Kubernets API URL中都有分组和版本信息，对于client端发送的请求和接受到的返回中的资源分组及版本都是确定的，因此在client端的序列化器没有版本转换的需求。 4: JSON序列化器是apimachinery中最为基础的序列化组件之一，官方开发者为了保证它的可靠性因此考虑到的情况也比较多。我们在这里会放弃一些严谨性，仅抽取一些关键步骤来大概说明。 5: 官方开发者用Negotiated(协商)命名此核心接口是因为这个序列化器通常在被用于Server端时可以执行HTTP内容协商（HTTP Content negotiation）。 HTTP内容协商是指在返回资源请求时如何选择特定的数据表示形式（尤其是客户端在请求资源时没有通过HTTP的Accept头明确指定接受格式时）。这使得上层封装的序列化器需要支持/集成多种数据类型的编/解码器。

    除此以外，在kubernetes Server端组件，例如apiserver中，需要将对资源的不同版本的请求通通转换为一个内部版本（APIInternalVersion或者也被称作HubVersion）以便对不同版本的资源统一进行相关逻辑的处理。这种资源版本在server端的转化也可以理解成一种server内部对资源版本的协商。🌧️ 我们可以找到官方开发者对negotiatedSerializerWrapper的这样一段注释：

    TODO: We should split negotiated serializers that we can change versions on from those we can change serialization formats on

    意思是说"我们应该将可以协商（资源）版本的序列化器与可以协商序列化格式的序列化器分开"。在官方开发者看来，资源版本的内部转换也可以称之为协商。🌧️

    因此，NegotiatedSerializer中所谓的协商可视为两个层次: i) 数据格式 ii) 资源内部版本转换 6: 通常我们所说的序列化器或者编解码器是指字节序列（[]bytes）与Go对象之间的转化，此处的"编解码器"并不能归为此类，它的作用是HTTP URL参数（Query Parameters）与Go对象之间的转换。 7: 注意此处的ListOptions与我们在之前再识kind小节中提及的DeleteOptions联系于区别：DeleteOptions与ListOptions同属于kind第三种类，但是DeleteOptions对应的是HTTP请求体而不是HTTP URL参数。↩
